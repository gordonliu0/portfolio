## Naming Analysis

Scope: `app/`, `components/`, `mdx-components.tsx`. Prioritized by how much the name misleads or breaks consistency.

### Critical — name fights intent or breaks domain consistency

**1. `PostPage` — `app/projects/[slug]/page.tsx:17`**
The default export on the project detail page is `PostPage`, but this is a *project* page — the sibling list is `ProjectsPage`, the data type is `Project`, the route is `/projects/[slug]`. "Post" is from a different domain.
→ `ProjectPage`

**2. `ReportView` / `view.tsx` — `app/projects/[slug]/view.tsx:5`**
Reads as "a view of a report." It actually fires a pageview beacon. The word "view" inverts with the component returning `null` and never rendering anything viewable.
→ `TrackPageview` (component), `track.tsx` or `pageview.tsx` (file)

**3. `/api/incr` + `POST /incr` — `app/api/incr/route.ts`**
`incr` leaks the Redis mechanism (`redis.incr`) into a public HTTP path. Callers think in terms of "record a pageview," not "increment a counter." Also not greppable.
→ `/api/pageview` (or `/api/views`), with the body still carrying `slug`

### High — generic or misleading at call sites

**4. `Page` component — `components/Page.tsx:3`**
Name is too generic to grep for, and collides with Next.js's own "Page" concept (`page.tsx`, `PostPage`, `ProjectsPage`). It's actually a two-column layout with a sidebar slot.
→ `ArticleShell` or `SidebarLayout`

**5. `gutter` prop — `components/Page.tsx:4,13`**
"Gutter" is a CSS-grid term describing *how* the slot is laid out, not *what* goes in it. Callers pass a back-link / aside content.
→ `aside` or `sidebar`

**6. `sorted` — `app/projects/page.tsx:8`**
Variable is filtered *and* sorted; name only covers half. Readers have to re-check the chain to see `published` filtering happened.
→ `visibleProjects` (or split `published` + `sortedByDate`)

**7. `nav` array + `link` key — `components/NavBar.tsx:6,24-30`**
Two issues: `nav` (singular) shadows the `<nav>` element rendered three lines below, and item shape uses `{ label, link }` where HTML convention (and the anchor `href` prop it feeds) is `href`.
→ `navItems`, and `link` → `href`

### Medium — asymmetry and low searchability

**8. `ascending` — `app/projects/_data.ts:36`**
Ascending *what*? It's projects sorted by date, used only to compute a stable ID. Name strips the subject.
→ `byDateAsc` or inline since it's only used once

**9. `topic` iterator over `interests` — `app/page.tsx:52`**
Collection is `interests`, iterator calls each one `topic`. Small break in the plural/singular symmetry rule.
→ `interest`

**10. `s` iterator over `socials` — `app/page.tsx:65`**
`s.href`/`s.value`/`s.label` reads as cryptic at three call sites. Single-letter is fine for trivial maps, but this one has branches (`s.href.startsWith("http")`).
→ `social`

### Minor — acceptable but notable

- `buf` in `app/api/incr/route.ts:26` — fine in scope; `digest` would be more semantic and matches the method.
- `mod` in `_data.ts:32,52` — fine; idiomatic abbreviation for "module."
- `rows` in `header.tsx:25` — fine; `metaRows` if you wanted to disambiguate.
- `ProjectMeta` vs `Project` — readable, though `Project = ProjectMeta & {slug,id}` makes `Project` the richer type, which is fine but worth noting the `Meta` suffix implies "about" rather than "subset."

### Summary of top 3 to fix first

1. `PostPage` → `ProjectPage` (consistency with the rest of the domain)
2. `ReportView` / `view.tsx` → `TrackPageview` / `track.tsx` (name inverts intent)
3. `/api/incr` → `/api/pageview` (mechanism leaking into public URL)
