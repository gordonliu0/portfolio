# Naming Analysis — wv7 Portfolio

## Critical

**1. `ReportView` / `view.tsx` (`app/projects/[slug]/view.tsx:5`)**
The component's job is to POST a pageview increment — it doesn't render anything and doesn't "report a view" in any domain sense. It's a side-effect component.
- **Problem**: Name describes how (sends a report) without naming *what* (pageview tracking). `view.tsx` collides conceptually with `view`=render.
- **Suggest**: `TrackPageview` component in `track-pageview.tsx`. Symmetric with the Redis key `pageviews:projects:<slug>`.

**2. `app/api/incr/route.ts`**
The route name `incr` leaks the Redis primitive (INCR command) into the public URL. Callers shouldn't know the storage engine.
- **Problem**: Names the implementation, not the intent. Not searchable as a domain term.
- **Suggest**: `app/api/pageviews/route.ts` (POST increments). Matches the `pageviews:*` Redis key family already in use.

**3. `components/Page.tsx` + `gutter` prop (`components/Page.tsx:3`)**
`Page` is too generic to grep and overlaps with Next.js's own `page.tsx` concept. The `gutter` prop names a CSS layout slot, not its role.
- **Problem**: Generic name (`Page`), and `gutter` describes position, not purpose (it holds a back-link / side nav).
- **Suggest**: Rename to `ArticleLayout` (or `ProjectLayout`) with prop `sidebar` or `backLink`.

## Major

**4. `PostPage` for a project page (`app/projects/[slug]/page.tsx:17`)**
The route renders a **project**, not a post. Inconsistent with the surrounding `Project`, `getProject`, `/projects/` vocabulary.
- **Suggest**: `ProjectPage`.

**5. `nav` array vs `NavBar` component (`components/NavBar.tsx:6`)**
The array item field is `link` but holds an href path string — `link` usually means an anchor element.
- **Suggest**: `{ label, href }` — matches the `socials` array in `app/page.tsx:3` (`href`) and standard web vocabulary. Also rename the array to `navItems` for searchability.

**6. `socials` and `s` (`app/page.tsx:3`, `:65`)**
`socials` is fine; the `s` alias in the `.map((s) => …)` is too terse for a name that crosses ~15 lines with repeated member access (`s.label`, `s.href`, `s.value`).
- **Suggest**: `.map((social) => …)`.

**7. `interests` vs heading "Currently" (`app/page.tsx:9`, `:49`)**
The list is labelled "Currently" in the UI but the const is `interests`. Two names for one concept.
- **Suggest**: Pick one. `currentFocus` or `nowList` if the "Currently" framing matters; otherwise rename the heading.

**8. `_data.ts` private module (`app/projects/_data.ts`)**
Underscore prefix is a Next.js convention to opt a folder out of routing, but for a single file it's just noise. The file exports the public `Project` type and loaders.
- **Suggest**: `projects.ts` or `content.ts`. Reserve `_` for folders Next.js actually treats specially.

**9. `getAllSlugs` + `getAllProjects` + `getProject` asymmetry (`_data.ts:21,28,45`)**
`getAllSlugs` is sync, `getAllProjects` is async. Fine, but `getProject(slug)` internally calls `getAllProjects()` and then filters — the name suggests a direct lookup.
- **Suggest**: Keep names, but if performance matters, consider `findProjectBySlug`. At minimum the asymmetry of sync/async `getAll*` functions is worth a second look.

## Minor

**10. `mod` for MDX module (`_data.ts:32,52`)**
`mod` is a common abbreviation but ambiguous (modulo? modify? module?). `mdxModule` or just the destructured `{ metadata, default: Component }` would be clearer.

**11. `rows` in `header.tsx:25`**
Local, short-lived — fine as-is. No change.

**12. `buf` / `hash` (`route.ts:26-32`)**
`buf` is fine for a local `ArrayBuffer`, but `hash` below is a hex string, not a hash buffer. Consider `hashHex` for precision.

**13. Redis key `deduplicate` (`route.ts:35`)**
Key prefix reads as a verb. Redis keys throughout are nouns (`pageviews:...`). Suggest `dedupe:<hash>:<slug>` or `viewed:<hash>:<slug>`.

**14. `ProjectMeta` vs `Project` (`_data.ts:5,17`)**
Clean split, but note `Project = ProjectMeta & { slug, id }` — "meta" is a weak modifier. Consider `ProjectFrontmatter` (what it literally is — MDX frontmatter) and `Project` for the enriched version. Domain-accurate.

**15. `formatHost` vs `formatDate` (`header.tsx:8,16`)**
Both return strings; symmetric naming is good. No change.

## Summary

Top priorities, in order:
1. Rename `ReportView` → `TrackPageview` and `/api/incr` → `/api/pageviews` (these leak implementation and mislead readers most).
2. Rename `components/Page.tsx` → `ArticleLayout` and `gutter` → `sidebar`/`backLink`.
3. `PostPage` → `ProjectPage`; unify `link` → `href` in the nav array; drop underscore from `_data.ts`.
