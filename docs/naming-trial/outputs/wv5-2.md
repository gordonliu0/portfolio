# Naming Analysis Results

Based on a full codebase scan, here are the most impactful naming issues grouped by severity. I verified file paths against the repo.

---

## CRITICAL — Misleading or confusing

**1. `app/projects/[slug]/view.tsx:5` — `ReportView`**
Sends a POST to `/api/incr` and returns `null`. Name implies a visual component.
→ `PageViewTracker`

**2. `app/projects/_data.ts:36-42` — `ascending`**
Temporary sorted-by-date array used only to build an id map; the returned `metas` is unsorted.
→ `projectsByDateAsc` (or inline it)

**3. `app/projects/[slug]/page.tsx:17` — `PostPage`**
Renders a *project*, not a post. Portfolio has no blog.
→ `ProjectPage`

**4. `components/Page.tsx:4` — `gutter` prop**
A CSS/typography term used for sidebar content. Leaks layout jargon into the component API.
→ `sidebar` or `aside`

---

## MAJOR — Inconsistency or vague in wide scope

**5. `app/projects/_data.ts` — `meta` / `metas` / `Component`**
Mixed casing and abbreviations in the same module (`meta: Project`, `Component: ComponentType`). Pick one style.
→ `{ metadata, Component }` — keep capitalized for the component, spell out `metadata`.

**6. `app/projects/_data.ts:33` — `mod`**
Imported MDX module. `mod` is unsearchable and ambiguous (modulo? modification?).
→ `projectModule`

**7. `app/projects/[slug]/header.tsx:3` — `Props`**
Anonymous `Props` type. Fine in one-component files, but the project uses `HeaderProps`/`ProjectHeaderProps` style elsewhere worth matching.
→ `HeaderProps`

**8. `app/projects/[slug]/header.tsx:24` — `Header`**
Collides semantically with the site/page header in `components/`. Project-specific header deserves a specific name.
→ `ProjectHeader`

**9. `components/NavBar.tsx:6` — `nav`**
Array of `{ label, link }`. Collides with the HTML `<nav>` element concept.
→ `navItems`

**10. `app/page.tsx:65` — iterator `s`**
Single-letter for a structured social object, inconsistent with `topic`/`project` used nearby.
→ `social`

**11. `app/projects/_data.ts:14` + `app/projects/page.tsx:9` — `published?: boolean` + `.filter((p) => p.published !== false)`**
Double-negative filter with defaults-to-true-when-undefined behavior is hard to read.
→ Rename to `draft?: boolean` (opt-in hide, positive check) **or** `isPublished` with an explicit default.

---

## MINOR — Local style/convention nits

**12. `app/projects/[slug]/header.tsx:25` — `rows`**
Array of label/value pairs rendered as a definition list, not a table. "Rows" encodes a layout assumption.
→ `fields` or `entries`

**13. `app/api/incr/route.ts:30-32` — `buf`, `hash`**
`buf` is the SHA-256 digest bytes; `hash` is its hex string. Names don't distinguish them.
→ `digest` (bytes) and `digestHex` (string)

**14. `app/projects/_data.ts:19` — `PROJECTS_DIR`**
SCREAMING_SNAKE for a value computed at runtime via `process.cwd()`. Other module-level constants in the file aren't styled this way.
→ `projectsDir`

**15. `app/page.tsx:3,9` — `socials`, `interests`**
Plurals of objects rather than of primitives. Minor, but `socialLinks` / `currentInterests` read more precisely.

---

## Recommended order of fixes
1. `ReportView` → `PageViewTracker` (fixes a genuinely misleading name)
2. `PostPage` → `ProjectPage` (file/export mismatch)
3. `meta`/`metas`/`mod` cleanup in `app/projects/_data.ts` (highest-leverage module, touched by every project page)
4. `gutter` → `sidebar` in the shared `Page` component (public API of a reusable component)
5. Everything else as you touch the surrounding code.

Want me to apply any of these? I'd recommend starting with #1–#4.
