# Naming Analysis

## Critical

### 1. `ReportView` — misleading component name
**File:** `app/projects/[slug]/view.tsx:5`

The component is named `ReportView` but it neither reports nor views. It's a side-effect-only client component that POSTs a pageview and returns `null`. "Report"/"View" together strongly imply a visual report.

- **Intent violated:** name describes nothing the component does for callers.
- **Suggest:** `TrackPageview` (or `RecordPageview` / `PageviewPing`).
- **Rename file too:** `view.tsx` → `track-pageview.tsx`. `view.tsx` is one of the least searchable file names possible.

### 2. `PostPage` — wrong domain noun
**File:** `app/projects/[slug]/page.tsx:17`

Function is named `PostPage`, but the surrounding system uses **project** everywhere (`Project` type, `getProject`, `/projects`, `content/projects`). This is a leftover from a blog-style template.

- **Consistency violated:** same concept, two names.
- **Suggest:** `ProjectPage`.

### 3. `/api/incr` — leaks implementation
**File:** `app/api/incr/route.ts` (route folder name)

`incr` is the Redis command used internally. Callers don't care how pageviews are stored — they care that they're being recorded. The route name should describe intent, not the storage primitive.

- **Intent violated:** "how" leaks into the public URL; renaming Redis → another store would force a URL change.
- **Suggest:** `/api/pageviews` or `/api/views`.

## Moderate

### 4. `Page` component collides with Next.js conventions
**File:** `components/Page.tsx`

In a Next App Router project, "Page" already means a route module. This `Page` component is actually a two-column layout wrapper (gutter + content). The overloaded name is also ungreppable.

- **Searchability + convention:** a grep for `Page` gets swamped.
- **Suggest:** `GutterLayout` or `PageLayout`. Rename the file to match.

### 5. `nav` const shadows the `<nav>` element
**File:** `components/NavBar.tsx:6`

`const nav = [...]` is the data, and `<nav>` is the JSX element below it. Also too generic.

- **Suggest:** `navItems` or `navLinks`.

### 6. `ascending` for a sorted array reads as an adjective, not a collection
**File:** `app/projects/_data.ts:36`

`const ascending = [...metas].sort(...)` — the name describes the sort direction, not the contents. Reader must trace back to see it's still `ProjectMeta[]`.

- **Suggest:** `byDateAsc`, `oldestFirst`, or `chronological`.

### 7. `socials` includes non-social entries
**File:** `app/page.tsx:3`

The array contains Email alongside GitHub/Instagram. Email isn't a "social" by most definitions.

- **Suggest:** `contacts` or `links`. And `socials.map((s) => ...)` at line 65 → `contacts.map((contact) => ...)` once renamed.

### 8. `interests` / `topic` mismatch
**File:** `app/page.tsx:9, 52`

`interests.map((topic) => ...)` — the singular of `interests` is `interest`, not `topic`. Minor symmetry break.

- **Suggest:** `interests.map((interest) => ...)`.

## Minor

### 9. `mod` for dynamic imports
**File:** `app/projects/_data.ts:32, 52`

`mod` is short but common idiom for dynamic `import()` results. Acceptable; flag only if you're tightening conventions — could be `mdxModule`.

### 10. `gutter` prop
**File:** `components/Page.tsx:4`

"Gutter" is typographically correct but the slot is effectively a sidebar/aside holding navigation. A web-domain reader would expect `aside` or `sidebar`. Keep if you want the typographic feel of the design system; otherwise `aside`.

### 11. `rows` in `Header`
**File:** `app/projects/[slug]/header.tsx:25`

Generic but accurate for `<dl>` rows. Fine as-is; `detailRows` if you want to be explicit.
