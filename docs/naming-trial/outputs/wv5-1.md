I've reviewed the full codebase (app/, components/, mdx-components.tsx). Here's the naming analysis, ordered by severity.

---

## Critical — names that mislead about behavior

### 1. `ReportView` component — doesn't render a view
**File:** `app/projects/[slug]/view.tsx:5`
The suffix `View` strongly implies UI, but the component returns `null` and only fires a fetch side-effect. A reader will grep for `ReportView` expecting rendered content.
**Suggest:** `TrackPageview` (or `RecordPageview`). Rename file to `track-pageview.tsx`.

### 2. `PostPage` — wrong domain term
**File:** `app/projects/[slug]/page.tsx:17`
The function is named `PostPage` but this route is for *projects*, not posts. There's no "post" concept anywhere else in the codebase.
**Suggest:** `ProjectPage`.

### 3. `/api/incr` route — opaque verb, no object
**File:** `app/api/incr/route.ts` + `view.tsx:7`
`incr` tells the reader *how* (the Redis primitive) instead of *what*. Route names should describe the resource/action.
**Suggest:** `/api/pageviews` (POST increments), or `/api/projects/view`. This also matches the `"pageviews"` Redis key prefix.

---

## High — generic or ambiguous names in public surfaces

### 4. `Page` component is too generic
**File:** `components/Page.tsx:3`
`Page` collides conceptually with Next.js' page primitive. The component is really a layout: a sticky sidebar slot + main content.
**Suggest:** `SidebarLayout` or `ArticleLayout`.

### 5. `gutter` prop — wrong metaphor
**File:** `components/Page.tsx:4`
A "gutter" is whitespace between columns; this prop takes actual content (a back-link) rendered in an `<aside>`. The name describes the space, not what fills it.
**Suggest:** `sidebar` or `aside`.

### 6. `link` field in nav items
**File:** `components/NavBar.tsx:7`
`{ label, link }` — `link` is ambiguous (element? URL?) and breaks the web convention of `href`. Everywhere else in the codebase `href` is used.
**Suggest:** `href`.

### 7. `socials` includes a non-social
**File:** `app/page.tsx:3`
`socials` contains Email, which isn't a social. The section is labeled "Reach" in the UI.
**Suggest:** `contacts` or `reachLinks` to match the UI label and the actual contents.

---

## Medium — clarity wins in local scope

### 8. `s` in `socials.map((s) => …)`
**File:** `app/page.tsx:65`
Single-letter param isn't idiomatic for objects with multiple fields. Reader has to peek at the source to know what `s.label` is.
**Suggest:** `contact` (after renaming #7) or `social`.

### 9. `nav` variable shadows semantic meaning
**File:** `components/NavBar.tsx:6`
`const nav = […]` next to `<nav>` element creates overlap; the const is the *data*, the element is the landmark.
**Suggest:** `navLinks` or `navItems`.

### 10. `active` → `isActive`
**File:** `components/NavBar.tsx:25`
Project convention (and CLAUDE.md naming rule) is `is`/`has` prefix for booleans.

### 11. `sorted` in projects list
**File:** `app/projects/page.tsx:8`
The variable is both filtered and sorted; `sorted` tells only half the story.
**Suggest:** `visibleProjects` or `publishedByDateDesc`.

### 12. `ascending` holds projects, not an order
**File:** `app/projects/_data.ts:36`
Reads like a boolean or enum; it actually holds the chronologically-sorted list used for ID assignment.
**Suggest:** `chronological` or `byDateAsc`.

### 13. `rows` in the project header
**File:** `app/projects/[slug]/header.tsx:25`
Generic; this is metadata entries, not table rows in any data sense.
**Suggest:** `metaRows` or `metadataEntries`.

### 14. Terminology drift: `views` vs `pageviews`
**Files:** `header.tsx:5`, `page.tsx:29`, `route.ts:43`
Redis key is `pageviews:projects:<slug>`; prop and local are `views`. Pick one term.
**Suggest:** `pageviews` throughout (matches the stored key and is unambiguous).

---

## Minor — stylistic / acceptable as-is

- **`mod`** in dynamic imports (`_data.ts:32,52`) — short scope, fine, but destructuring `{ default: Component, metadata }` reads better.
- **`metas`** (`_data.ts:30`) — awkward plural; `projectMetas` is only marginally better.
- **`buf`** (`route.ts:26`) — short-scoped crypto local; acceptable.
- **`topic`** for interests (`app/page.tsx:52`) — readable; fine.
- **`Home`** default export — Next.js convention; fine.
- **`formatHost` / `formatDate`** — clear; fine.

---

## Recommended order of attack

1. **#1–3 first** — these genuinely mislead readers.
2. **#4–7 next** — they're on the public surface (component API, nav config, content).
3. **#8–14** — tidy-up during the next touch of each file.
4. Minor items — leave alone unless you're already editing.

Happy to apply any subset — say which numbers.
